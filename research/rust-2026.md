# Rust en production en 2026 : 5 cas d'usage concrets

Cette note rassemble cinq domaines représentatifs de l'écosystème Rust tel qu'il se présente en 2026, à partir de recherches web récentes (voir sources en fin de document). Pour chaque domaine : le contexte, un extrait de code illustratif (écrit pour cette note, inspiré des patterns observés mais non recopié d'un dépôt existant), les avantages et les inconvénients.

## 1. Infrastructure cloud et virtualisation légère (microVMs, proxys réseau)

### Contexte
AWS a construit **Firecracker**, la technologie de microVM qui alimente Lambda et Fargate, entièrement en Rust. Plus largement, les proxys réseau haute performance et les couches d'isolation d'infrastructure cloud (load balancers, sidecars, VMM) s'appuient sur Rust pour combiner performance proche du C et sécurité mémoire, dans un contexte où une seule faille de sécurité dans l'hyperviseur compromet l'isolation entre locataires.

### Extrait de code illustratif
```rust
use tokio::io::{copy_bidirectional, AsyncWriteExt};
use tokio::net::{TcpListener, TcpStream};

async fn handle_conn(mut inbound: TcpStream, upstream_addr: &str) -> std::io::Result<()> {
    let mut outbound = TcpStream::connect(upstream_addr).await?;
    // Relaie le trafic dans les deux sens sans copie intermédiaire superflue.
    match copy_bidirectional(&mut inbound, &mut outbound).await {
        Ok((from_client, from_upstream)) => {
            tracing::info!(from_client, from_upstream, "connexion terminée proprement");
        }
        Err(e) => tracing::warn!(error = %e, "connexion interrompue"),
    }
    outbound.shutdown().await
}

pub async fn run_proxy(listen_addr: &str, upstream_addr: String) -> std::io::Result<()> {
    let listener = TcpListener::bind(listen_addr).await?;
    loop {
        let (socket, _) = listener.accept().await?;
        let upstream = upstream_addr.clone();
        tokio::spawn(async move {
            if let Err(e) = handle_conn(socket, &upstream).await {
                tracing::error!(error = %e, "erreur de proxy");
            }
        });
    }
}
```

### Avantages
- Pas de garbage collector : latence prévisible, essentielle pour des proxys et hyperviseurs qui doivent tenir des SLA de queue de latence stricts.
- Le modèle de propriété (*ownership*/*borrowing*) élimine à la compilation des classes entières de bugs (use-after-free, data races) critiques dans du code qui gère l'isolation entre locataires cloud.
- L'écosystème async (Tokio) permet de gérer des dizaines de milliers de connexions concurrentes avec une empreinte mémoire réduite, comparable à Go mais sans runtime GC.
- Binaires statiques, démarrage quasi instantané : atout pour des composants d'infrastructure invoqués à très haute fréquence (microVMs Firecracker démarrant en quelques millisecondes).

### Inconvénients
- Le code réseau async en Rust reste verbeux et la courbe d'apprentissage (lifetimes, `Pin`, traits async) ralentit l'onboarding d'équipes venant de Go ou Java.
- Le débogage de deadlocks ou de blocages dans un exécuteur async peut être plus difficile qu'avec des outils matures type pprof/goroutine dumps de Go.
- Écosystème de bibliothèques réseau encore fragmenté (Tokio vs async-std vs smol), ce qui complique l'interopérabilité entre crates.
- Pour des équipes ayant déjà une expertise Go et un besoin de vélocité produit plus que de performance extrême, Go reste souvent un choix plus rapide à livrer.

## 2. WebAssembly côté edge (fonctions serverless, edge computing)

### Contexte
En 2026, WebAssembly est devenu un runtime de production à part entière : 67 % des équipes interrogées l'utilisent en production (contre 47 % en 2024), et les déploiements côté serveur/edge dépassent désormais les usages purement navigateur. Cloudflare Workers, Fastly Compute et des runtimes comme Wasmtime, Spin et WasmEdge exécutent massivement du code compilé depuis Rust, avec des cold starts de l'ordre de 1 à 5 millisecondes.

### Extrait de code illustratif
```rust
use spin_sdk::http::{IntoResponse, Request, Response};
use spin_sdk::http_component;

#[http_component]
fn handle_edge_request(req: Request) -> anyhow::Result<impl IntoResponse> {
    let country = req
        .header("cf-ipcountry")
        .and_then(|v| v.as_str())
        .unwrap_or("XX");

    let body = format!(r#"{{"greeting":"bonjour","edge_region":"{country}"}}"#);

    Ok(Response::builder()
        .status(200)
        .header("content-type", "application/json")
        .body(body)
        .build())
}
```

### Avantages
- Compilation vers `wasm32-wasi` avec un binaire compact et sandboxé par défaut : modèle de sécurité idéal pour exécuter du code tiers non fiable sur des centaines de points de présence.
- Démarrage à froid quasi nul (pas de JVM, pas d'interpréteur), ce qui rend Rust+Wasm particulièrement adapté à la facturation à l'exécution des plateformes serverless/edge.
- Le même binaire Wasm peut cibler navigateur, serveur et edge (WASI Preview 2 stabilisé en 2026), réduisant la duplication de logique métier.
- Absence de GC : pas de pauses de collecte imprévisibles dans un environnement où chaque milliseconde de latence est facturée ou mesurée en SLA.

### Inconvénients
- L'écosystème WASI est encore en mouvement (Preview 2 tout juste stabilisé) : certaines API systèmes (threads, sockets bruts) restent limitées ou expérimentales.
- Le débogage d'un module Wasm en production (stack traces, profiling) est moins mature que pour un binaire natif classique.
- Toutes les crates Rust ne sont pas compatibles `no_std`/Wasm (accès disque, threads natifs), ce qui impose parfois de réécrire des dépendances.
- Pour des workloads qui n'ont pas besoin de sandboxing multi-tenant ni de portabilité extrême, un simple binaire natif Rust (ou même une fonction Node.js) reste plus simple à opérer.

## 3. Systèmes embarqués et automobile critique en sécurité

### Contexte
L'adoption de Rust dans l'automobile est passée de 0,3 % des organisations en 2024 à environ 5,3 % en 2025, avec des cas de production concrets (ECU du Volvo XC90, Polestar 3). Des toolchains Rust certifiées ISO 26262 (Ferrous Systems, HighTec associé aux microcontrôleurs Stellar de STMicroelectronics, certifiés ASIL D) permettent désormais de qualifier du code Rust pour des composants de sécurité critique, en particulier sur des périmètres ASIL B/C.

### Extrait de code illustratif
```rust
#![no_std]
#![no_main]

use rtic::app;

#[app(device = stm32_hal::pac, dispatchers = [USART1])]
mod app {
    use stm32_hal::gpio::{Output, PushPull, PA5};

    #[shared]
    struct Shared {
        brake_signal_active: bool,
    }

    #[local]
    struct Local {
        status_led: PA5<Output<PushPull>>,
    }

    #[init]
    fn init(cx: init::Context) -> (Shared, Local) {
        // Initialisation matérielle minimale, sans allocation dynamique.
        (Shared { brake_signal_active: false }, Local { status_led: cx.local.led })
    }

    #[task(binds = TIM2, shared = [brake_signal_active], priority = 3)]
    fn on_brake_sensor_tick(mut cx: on_brake_sensor_tick::Context) {
        cx.shared.brake_signal_active.lock(|active| {
            *active = read_sensor_safely();
        });
    }

    fn read_sensor_safely() -> bool {
        // Logique métier volontairement simplifiée pour l'exemple.
        true
    }
}
```

### Avantages
- `#![no_std]` + vérification des emprunts à la compilation : élimine à la source les corruptions mémoire typiques du C embarqué, sans coût de garbage collector incompatible avec le temps réel.
- Des frameworks comme RTIC ordonnancent les tâches par priorité avec des garanties statiques d'absence de data race — des études 2026 rapportent une réduction de 95 % des incidents de data race par rapport à des implémentations C équivalentes.
- Les toolchains désormais certifiées ISO 26262 (ASIL D pour le compilateur HighTec) lèvent l'obstacle réglementaire qui freinait l'automobile.
- Le futur Cyber Resilience Act européen (obligations à partir de 2027) pousse la sécurité mémoire comme critère d'achat, ce qui favorise mécaniquement Rust face au C/C++ historique.

### Inconvénients
- L'écosystème d'outils de certification, de debuggers et d'IDE reste bien moins mature que celui de C/C++ dans l'automobile (MISRA C, décennies d'outillage AUTOSAR).
- Le vivier d'ingénieurs embarqués formés à Rust est encore restreint comparé aux effectifs C/C++ des équipementiers.
- La qualification complète (ASIL D de bout en bout, pas seulement le compilateur) est encore en cours de maturation ; en 2026 l'adoption reste concentrée sur des composants à criticité plus faible (ASIL B/C).
- Pour du code legacy déjà qualifié et stable en C, une réécriture complète en Rust n'est pas toujours justifiable économiquement à court terme.

## 4. Moteurs de bases de données distribuées et de traitement de données

### Contexte
Rust s'est imposé comme langage de choix pour réécrire ou construire des moteurs de stockage et de traitement de données exigeants en fiabilité : **TiKV** (base clé-valeur transactionnelle distribuée, projet CNCF), **SurrealDB** (base multi-modèle temps réel pouvant s'appuyer sur TiKV comme moteur de stockage distribué), et **Polars** (moteur de DataFrame et requêtes paresseuses, alternative performante à pandas). Ces projets misent sur Rust pour offrir des garanties de robustesse en cas de crash et un parallélisme sûr à l'échelle du datacenter.

### Extrait de code illustratif
```rust
use std::collections::BTreeMap;
use std::sync::RwLock;

pub struct Region {
    id: u64,
    data: RwLock<BTreeMap<Vec<u8>, Vec<u8>>>,
}

impl Region {
    pub fn get(&self, key: &[u8]) -> Option<Vec<u8>> {
        self.data.read().expect("verrou en lecture empoisonné").get(key).cloned()
    }

    pub fn put(&self, key: Vec<u8>, value: Vec<u8>) -> Result<(), StorageError> {
        let mut guard = self.data.write().map_err(|_| StorageError::PoisonedLock)?;
        guard.insert(key, value);
        Ok(())
    }

    /// Découpe la région en deux quand elle dépasse un seuil de taille,
    /// pattern central du "region splitting" utilisé par les stores type TiKV.
    pub fn split_at(&self, split_key: &[u8]) -> Region {
        let mut guard = self.data.write().expect("verrou empoisonné");
        let right_half = guard.split_off(split_key);
        Region { id: self.id + 1, data: RwLock::new(right_half) }
    }
}

#[derive(Debug)]
pub enum StorageError {
    PoisonedLock,
}
```

### Avantages
- Fiabilité au crash : l'absence de comportements indéfinis et une gestion d'erreurs explicite (`Result`, pas d'exceptions silencieuses) réduisent le risque de corruption de données sur un chemin d'écriture critique.
- Performances proches du C++ pour du code intensif en I/O et en structures de données (arbres LSM, index), sans coût de GC qui introduirait des pauses lors de compactions ou de requêtes analytiques.
- `Send`/`Sync` vérifiés à la compilation : le parallélisme sur des moteurs multi-cœurs (comme le moteur de requêtes de Polars) est sûr par construction, ce qui est précieux pour du code de calcul distribué.
- Interopérabilité facile via FFI (bindings Python pour Polars, clients dans d'autres langages) grâce à une ABI C stable.

### Inconvénients
- La complexité d'implémentation de structures de données concurrentes performantes (lock-free, MVCC) en Rust demande une expertise pointue ; le compilateur protège contre certains bugs mais pas contre les erreurs de conception distribuée (cohérence, consensus).
- Temps de compilation longs sur de gros moteurs (TiKV, Polars) qui ralentissent le cycle itératif par rapport à des langages interprétés.
- Le monomorphisation intensive des generics peut faire gonfler la taille des binaires et allonger encore la compilation.
- Pour des besoins de prototypage rapide de logique métier au-dessus de la base (pas le moteur lui-même), un langage plus haut niveau reste souvent préférable ; Rust est surtout justifié pour le cœur du moteur, pas pour toute la pile applicative.

## 5. Outils en ligne de commande et tooling développeur

### Contexte
Rust a largement remplacé les utilitaires Unix historiques (grep, find, ls, cat) par des équivalents plus rapides et plus ergonomiques (ripgrep, fd, bat, eza). Le cas le plus emblématique en 2026 est celui d'**Astral** : **uv** (gestionnaire de paquets/environnements Python) et **Ruff** (linter/formatter Python), tous deux écrits en Rust, ont atteint une adoption massive (uv dépasse 126 millions de téléchargements mensuels sur PyPI, plus de 85 000 étoiles GitHub), au point qu'OpenAI a annoncé début 2026 l'acquisition d'Astral pour en faire un pilier de son outillage Codex.

### Extrait de code illustratif
```rust
use clap::Parser;
use rayon::prelude::*;
use std::path::PathBuf;

#[derive(Parser)]
#[command(name = "quicklint", about = "Linter minimal inspiré de Ruff")]
struct Cli {
    /// Fichiers ou dossiers à analyser
    paths: Vec<PathBuf>,
    #[arg(long, default_value_t = false)]
    fix: bool,
}

fn main() -> anyhow::Result<()> {
    let cli = Cli::parse();

    let files = collect_source_files(&cli.paths)?;

    // Analyse chaque fichier en parallèle, un cœur par fichier.
    let issues: Vec<_> = files
        .par_iter()
        .flat_map(|path| lint_file(path).unwrap_or_default())
        .collect();

    for issue in &issues {
        println!("{}: {}", issue.location, issue.message);
    }

    if cli.fix {
        apply_autofixes(&issues)?;
    }

    std::process::exit(if issues.is_empty() { 0 } else { 1 });
}
```

### Avantages
- Démarrage instantané (pas de runtime à charger) et parallélisme trivial avec des crates comme `rayon` : un linter ou un résolveur de dépendances traite des milliers de fichiers en exploitant tous les cœurs disponibles, ce qui explique les gains de 10x à 100x observés face à des équivalents Python (Ruff vs flake8, uv vs pip).
- Distribution simple : un binaire unique par plateforme, sans dépendance à un interpréteur ou une VM du langage cible (précieux pour outiller un écosystème Python sans imposer un runtime Rust à l'utilisateur final).
- Sécurité mémoire et absence de crash sur des entrées malformées, important pour un outil exécuté des millions de fois par jour dans des CI.
- Écosystème de crates CLI mûr (`clap`, `serde`, `rayon`, `ignore`) qui accélère fortement le développement de ce type d'outil.

### Inconvénients
- Réécrire un outil existant (linter, résolveur) en Rust pour remplacer un équivalent dans le langage cible (ici Python) demande de dupliquer et maintenir la sémantique fine de l'écosystème d'origine (règles de style, résolution de versions), ce qui est un investissement d'ingénierie important.
- La compilation croisée multi-plateforme (Linux/macOS/Windows, x86/ARM) ajoute de la complexité à la chaîne de release par rapport à un simple paquet interprété.
- Pour un petit script interne à usage ponctuel, le coût d'entrée de Rust (compilation, gestion des types stricte) est disproportionné par rapport à un script Python ou shell rapide à écrire.
- La dépendance croissante de l'écosystème Python à des outils Rust externes (uv, Ruff) crée un couplage stratégique à une entreprise/organisation tierce (illustré par l'acquisition d'Astral par OpenAI), ce qui est un point de vigilance en gouvernance open source.
